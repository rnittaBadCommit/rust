## Checklist

### 1. 仕様と正しさ

- 入出力の仕様を満たしているか
- 正常系だけでなく、空入力、最小値、最大値、境界値を扱えているか
- 問題文やコメントの意図と実装がずれていないか
- 条件分岐や `match` に抜け漏れがないか

### 2. 型の使い方

- 所有権が不要なのに `String` や `Vec<T>` を受け取っていないか
- `&T` / `&mut T` / `T` の選び方が自然か
- 空や失敗を番兵値でごまかさず、`Option` / `Result` を使えているか
- `bool`, `enum`, `struct` で意味が明確になっているか

### 3. 借用とライフタイム

- 不要な clone, to_string, collect が増えていないか
- 借用で済むところを所有にしていないか
- 可変借用と共有借用の切り分けが無理のない形か
- ライフタイム注釈が必要以上に複雑になっていないか

### 4. エラー処理

- 失敗しうる処理で `unwrap`, `expect`, `panic!` を雑に使っていないか
- `Result<T, E>` を返すべき箇所で自然にエラーを返しているか
- `?` を使うと見通しが良くなる箇所がないか
- エラー時の挙動が呼び出し側にとって予測しやすいか

### 5. API と責務

- 関数やメソッドが一つの責務に収まっているか
- 名前で役割が分かるか
- 戻り値の型が呼び出し側にとって扱いやすいか
- `impl Type` と `impl Trait for Type` の使い分けが自然か

### 6. 可読性

- 変数名、関数名、型名が目的を表しているか
- 長すぎる関数や深すぎるネストがないか
- コメントが不足しているか、逆にコードで表せることを説明しすぎていないか
- C の癖をそのまま持ち込んで、Rust で読みにくくなっていないか

### 7. Rust らしさ

- 手書きループより標準メソッドの方が自然な箇所がないか
- `match`, `if let`, iterator, pattern matching を素直に使えているか
- 標準ライブラリや既存 trait で表現できるものを自前実装しすぎていないか
- 正しい実装でも、標準ライブラリや既存メソッドでより簡潔かつ意図が伝わりやすく書けるなら、改善案として具体例を挙げる
- 可変状態を必要以上に広げていないか

### 8. テストと確認

- 期待される主要ケースのテストがあるか
- 境界条件や失敗ケースの確認があるか
- `cargo test`, `cargo check`, `cargo run` のどれで確認したか分かるか
- 演習なら「通った」だけでなく、なぜ通るか説明できるか

### 9. パフォーマンスとメモリ

一般的なアプリケーションレビュー向けの基本観点です。

- 明らかに不要な確保やコピーがないか
- 毎回線形探索しているなど、簡単に悪化する箇所がないか
- 学習段階として許容できる単純さと、避けるべき無駄を区別できているか
- データサイズが増えたときに破綻しないか

### 10. 高性能要求コード向け

kernel、ランタイム、割り込み周辺、allocator、scheduler、ホットパス、何度も呼ばれるライブラリなどでは、上の基本観点に加えて次も確認します。

- 呼び出し頻度の高い経路で不要なコピー、確保、初期化、ゼロ埋めをしていないか
- 毎回の線形探索、不要な境界チェック、不要な分岐、繰り返し計算を持ち込んでいないか
- データ構造の選択がアクセスパターンに合っているか
- キャッシュ局所性を悪化させるレイアウトやメモリアクセスになっていないか
- ホットパスで trait object, dynamic dispatch, boxing を使う必要が本当にあるか
- inlining, monomorphization, const 化で素直に消せるコストを残していないか
- syscall, lock, atomic, allocation のような高コスト操作が頻発していないか
- ロック粒度、競合、false sharing、メモリ順序の選び方に無理がないか
- パフォーマンス改善のために安全性や保守性を崩す場合、その trade-off が妥当か
- ベンチマーク、プロファイル、計測値なしに性能断定していないか

### 11. 学習観点

- Rust のどの概念を練習するコードなのかが明確か
- C と比べたときの違いを説明できるか
- コンパイラが何を守ってくれているか説明できるか
- 修正案が「なぜ Rust ではこちらが自然か」まで示せているか

## Short Review Prompts

- このコードに仕様上のバグはあるか
- 所有権、借用、`Option`, `Result` の使い方に無理はあるか
- `unwrap` や不要な clone はないか
- より自然な Rust の書き方はあるか
- テストは何が足りないか
- ホットパスや低レベル実装として見たとき、避けるべきコストはないか
