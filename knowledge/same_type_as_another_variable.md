# 別の変数と同じ型で宣言したい

Rust では、C の GNU 拡張 `typeof(x)` や C++ の `decltype(x)` のように、
「別の変数の型をその場で取り出して変数宣言に書く」書き方は、
stable Rust にはありません。

つまり、次のような発想です。

```rust
// こういう構文は stable Rust にはない
// let b: typeof(a) = 20;
```

## まずは型推論に任せる

いちばん自然なのは、型注釈を書かずに初期化式から推論させることです。

```rust
fn main() {
    let a: u32 = 10;
    let b = a + 20;

    // a も b も u32
    println!("{a} {b}");
}
```

`b` の型は `a + 20` から `u32` と推論されます。

## `: _` は「推論して」の意味

```rust
fn main() {
    let a: u32 = 10;
    let b: _ = a + 20;
}
```

この `: _` は、
「型を書かない代わりに、右辺から推論してほしい」という意味です。

ただし、これは「`a` と同じ型にして」という専用構文ではありません。
結果として右辺が `a` と同じ型になれば、同じ型になります。

## 同じ型名を何度も使うなら型エイリアス

何度も同じ型を書くなら、変数どうしを参照するのではなく、
型名に別名を付けるほうが Rust では自然です。

```rust
type UserId = u64;

fn main() {
    let a: UserId = 10;
    let b: UserId = 20;
}
```

## 将来、元の型が変わったらどうなるか

「別の変数と同じ型にしておけば、元の型変更に強いのでは」
と考えたくなりますが、Rust ではここでコンパイルエラーになるのは普通で、
むしろ望ましいことが多いです。

たとえば slice の長さや添字は `usize` という前提で多くの API ができています。
もし `len()` の戻り値型だけ別物になったら、
インデックス計算や `xs[i]` と合わなくなるので、どのみち見直しが必要です。

つまり:

- 型変更でコンパイルエラーになる
- その場所を直す

という流れは、Rust では「壊れた前提を早く見つける」ための正常な挙動です。

もし「同じ意味の型」を何か所にも書いていて、将来まとめて変えたいなら、
変数を参照するのではなく `type Index = usize;` のように
型エイリアスで表すほうが向いています。

## 「2つは同じ型であってほしい」はジェネリクスで表す

関数や構造体で
「この2つは同じ型」
という関係そのものを表したいなら、ジェネリクスを使います。

```rust
fn make_pair<T>(a: T, b: T) -> (T, T) {
    (a, b)
}

fn main() {
    let ok = make_pair(10, 20);
    // let ng = make_pair(10, "hello"); // 型が違うのでエラー
}
```

ここでの `a` と `b` は、同じ `T` なので同じ型でなければなりません。

## まとめ

- stable Rust に `typeof(expr)` のような構文はない
- 変数宣言では、まず型推論に任せるのが自然
- 同じ型名を繰り返すなら `type` エイリアスを使う
- 「同じ型であること」を表したいならジェネリクスを使う
