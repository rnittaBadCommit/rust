# 配列 `[T; N]` とスライス `[T]` / `&[T]`

Rust では、

- 配列 `[T; N]` は「`T` を `N` 個ちょうど持つ値」
- スライス `[T]` は「`T` が連続して並んだ領域そのものを表す型」
- 実際によく使うのは `&[T]` で、「その連続領域への借用」

です。

## まず結論

```rust
fn main() {
    let a: [i32; 4] = [10, 20, 30, 40];

    let whole: &[i32] = &a;
    let part: &[i32] = &a[1..3];

    println!("{a:?}");
    println!("{whole:?}");
    println!("{part:?}");
}
```

- `a` は配列。値を 4 個そのまま持つ
- `whole` は配列全体を見るスライス
- `part` は一部だけを見るスライス

`part` は新しい配列を作っていません。
元の `a` の一部を「ここからここまで」と参照しているだけです。

## C でたとえると

```c
int sum(const int *xs, size_t len);
```

Rust の

```rust
fn sum(xs: &[i32]) -> i32
```

は、感覚としてかなりこれに近いです。

- `const int *xs` に近いもの: 先頭要素への参照
- `size_t len` に近いもの: 長さ

つまり `&[i32]` は「ポインタ + 長さ」に近い見方でほぼ合っています。

## 何が違うのか

### 配列 `[T; N]`

- 値を所有する
- 長さ `N` は型の一部
- `[i32; 4]` と `[i32; 5]` は別の型

```rust
let a: [i32; 4] = [1, 2, 3, 4];
```

これは「`i32` を 4 個持つ値」です。
長さ 4 という情報がコンパイル時に確定しています。

### スライス `[T]`

- 所有しない
- 長さは実行時に決まる
- 単体の `[T]` はサイズ不定なので、普通は値としてそのまま持たない
- ふつうは `&[T]` や `&mut [T]` として使う

```rust
let a = [1, 2, 3, 4];
let s: &[i32] = &a[1..3];
```

`s` は `[2, 3]` というコピーを持っているわけではなく、
`a` の中の一部分を指しています。

## 「スライスはオーバーロードできる？」への答え

基本的には違います。

スライスは Rust に最初からある型です。
「オーバーロードできる型」ではなく、「標準で定義されている連続領域ビュー」です。

混乱しやすいのは `a[1..3]` の見た目です。
これは「範囲指定で一部を取り出す」文法に見えますが、
内部では添字演算に関する仕組みが使われています。

ただし、学習の最初の段階では、

- `a[1..3]` は「配列や `Vec` の一部をスライスとして見る」

と理解しておけば十分です。

## `let whole = &a;` としたときの型

```rust
let a = [10, 20, 30, 40];

let whole = &a;
let part = &whole[1..3];
```

これはできます。

- `whole` の型はまず `&[i32; 4]`
- `part` の型は `&[i32]`

です。

つまり

```rust
let part = &a[1..3];
```

と比べて、`part` 自体は同じスライスになります。

違うのは `whole` の型です。

- `&a` は「長さ 4 の配列への参照」
- `&a[..]` や `let whole: &[i32] = &a;` は「スライスへの参照」

```rust
let a = [10, 20, 30, 40];

let whole1 = &a;     // &[i32; 4]
let whole2 = &a[..]; // &[i32]
```

## 何が変わるのか

実行時には、ほぼ気にしなくてよいです。

- どちらも要素をコピーしない
- どちらも元の配列を借りて見るだけ
- `part` の内容も同じ

主に変わるのはコンパイル時の型です。

```rust
fn takes_slice(xs: &[i32]) {}
fn takes_array_4(xs: &[i32; 4]) {}

fn main() {
    let a = [10, 20, 30, 40];

    let whole1 = &a;     // &[i32; 4]
    let whole2 = &a[..]; // &[i32]

    takes_slice(whole1);
    takes_slice(whole2);

    takes_array_4(whole1);
    // takes_array_4(whole2); // これは不可
}
```

`whole1` は必要なら `&[i32]` としても使えますが、
`whole2` は最初からスライスなので `&[i32; 4]` が必要な場所には使えません。

## `&[i32]` と `&[i32; 4]` のサイズ

参照そのもののサイズだけを見ると、
ふつうは `&[i32]` のほうが大きいです。

- `&[i32; 4]` は「サイズ既知の型への参照」なので、基本はポインタ 1 個
- `&[i32]` は「スライスへの参照」なので、基本は `データ先頭ポインタ + 長さ`

64bit 環境なら、よくあるサイズ感はこうです。

- `&[i32; 4]`: 8 バイト
- `&[i32]`: 16 バイト

```rust
use std::mem::size_of;

fn main() {
    println!("{}", size_of::<&[i32; 4]>()); // 64bit なら 8
    println!("{}", size_of::<&[i32]>());    // 64bit なら 16
}
```

これは C でたとえると、

- `&[i32; 4]` は `int (*)[4]` に近い
- `&[i32]` は `int *ptr` と `size_t len` の組に近い

という違いです。

ただし、ここで増減するのは「参照の値」のサイズであって、
元の配列本体のサイズではありません。

```rust
let a = [10, 20, 30, 40];
```

この配列本体はどちらにしても `i32` 4 個分です。

また、最適化後の実際の機械語では、
参照がレジスタに入ったり、変数自体が消えたりするので、
「必ずそのぶんだけスタックを使う」とまでは言えません。
まずは「型としてのサイズは `&[i32]` のほうが大きい」と覚えるのが実用的です。

## API で `&[T]` がよく使われる理由

```rust
fn sum(xs: &[i32]) -> i32 {
    let mut total = 0;
    for x in xs {
        total += *x;
    }
    total
}

fn main() {
    let a = [1, 2, 3, 4];
    let v = vec![1, 2, 3, 4];

    println!("{}", sum(&a));
    println!("{}", sum(&v));
    println!("{}", sum(&v[1..3]));
}
```

`&[T]` を受け取る関数にしておくと、

- 配列全体
- `Vec` 全体
- 一部分

を同じ形で受け取れます。

`[T; N]` を引数にすると、長さごとに別の型になってしまい、使いにくいです。

## 一言でまとめると

- 配列 `[T; N]`: データ本体を持つ固定長の箱
- スライス `&[T]`: その箱や `Vec` の中の連続した範囲を借りて見る窓
