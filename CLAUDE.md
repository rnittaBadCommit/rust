## 「コミットをお願いします」と言われたとき

この依頼は「現在のワークツリーを確認し、適切な粒度で 1 個以上のコミットを作る」という意味として扱う。追加の確認は不要。まず自分で整理して進める。

1. `git status --short` と `git diff --stat` で全体像を把握する。
2. `git diff` と必要なら `git diff -- <path>` で変更の意図を確認する。
3. 次の基準でコミット単位を決める。
   - 同じ学習テーマ、同じ目的の変更はまとめる
   - 無関係な演習、`knowledge/` のノート、運用ファイルは分ける
   - 新規 workspace 追加は、既存演習の小修正とは分ける
   - 直接対応する説明文や README は、そのコード変更と一緒でよい
   - 迷ったら小さめに分ける
4. すでに stage されている変更があれば、利用者の意図の手がかりとして扱う。ただし明らかに混在していれば整理し直してよい。
5. stage 内容を確認してからコミットする。必要なら部分的に stage して分ける。
6. 1 つコミットしたら `git status --short` を見直し、残りの変更があれば次のコミットに進む。
7. 最後に、作成した各コミットの hash と message、残した変更があればその理由を短く報告する。

## 追加ルール

- 依頼がコミットだけなら、コードレビューや追加実装を始めない。主目的は既存変更を整理してコミットすること
- コミット時に `cargo check` / `cargo test` などのビルド・テスト確認は不要。変更をそのまま整理してコミットする
- コミット時の `git add` / `git commit` / `git status` / `git diff` など、`rust_study` 配下のファイルに対する非破壊的な Git 操作は、ユーザーへの許可確認なしで実行してよい。ただし `git reset --hard` やファイル削除などの破壊的操作は、必要性を示したうえで許可を得ること
- 利用者が追加・更新した関連ファイルも、同じ論理単位ならコミット対象に含めてよい
- 空コミットは作らない
- 分離が不確実な変更は無理に混ぜず、分けられる分だけコミットして残りを報告する
- `git reset --hard` や不用意な `git checkout --` など、破壊的な Git 操作はしない
